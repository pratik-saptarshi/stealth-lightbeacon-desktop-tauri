P1 Monolithic Violation of SRP in Rust Subsystem
src-tauri/src/lib.rs
1. Narrative Description

A single file `lib.rs` spans over 3,026 lines and handles all Tauri commands, thread synchronization, process lifecycle management, cleartext serialization, HTTP ureq/reqwest clients, local scanning functions, and mock integrations. This is a severe violation of the Single Responsibility Principle. Modular modules inside `src-tauri/src/commands/` and `src-tauri/src/domain/` exist but are completely empty and unused, indicating a decoupled architecture was planned but entirely abandoned.

2. Code Evidence
Monolithic Rust File Structure: 
// src-tauri/src/lib.rs (Line 1 to 3026)
use std::{collections::HashMap, env, fs, path::{Path, PathBuf}, process::{Child, Command, Stdio}, sync::Mutex};

// Structs, serialization models, state management, HTTP requests, helper utils, 
// and 11 separate #[tauri::command] handlers all crowded in one flat file...
3. Raised By Reviewers
Reviewer Assessed Severity Comment / Rationale
Architecture Critic P1 Makes maintenance highly complex and prevents mock integration units.
Correctness Hawk P2 Creates higher locking dependency overlaps under async calls.
4. Judge Ruling

Verdict: Upheld as P2 severity (Adjusted downward from P1 by Judge in Phase 11). While refactoring monolithic files is a critical maintainability concern, it does not cause immediate runtime errors. Extracting Tauri command handlers to dedicated files is required before modular capability features are introduced.

5. Fix Recommendation

Move commands to modular closures inside the empty `src-tauri/src/commands/` directory:

// Extract commands:
// src-tauri/src/commands/evaluations.rs
use tauri::{State, AppHandle};
use crate::{AppState, BackendConfig, ApiError};

#[tauri::command]
pub async fn create_evaluation(state: State<'_, AppState>) -> Result {
    // isolated command logic
}

P1 AppState Lock Contention & Deadlock Risk
src-tauri/src/lib.rs
1. Narrative Description

`AppState` defines 5 granular `Mutex` fields to lock different variables individually. In Tauri async contexts, multiple concurrent commands attempt to obtain these guards sequentially without any unified ordering contract. For example, `set_backend_config` locks the backend state inside `stop_local_backend_in_app_state` and then replaces backend configurations, locking config variables. Concurrent polling checks perform this in reverse order, which represents a classical thread deadlock path.

2. Code Evidence 
// src-tauri/src/lib.rs:239
struct AppState {
    backend_config: Mutex,
    bootstrap_error: Mutex>,
    local_backend: Mutex>,
    standalone_jobs: Mutex>,
    standalone_sequence: Mutex,
}
4. Judge Ruling

Verdict: Upheld as P1 [EXISTING_DEFECT]. The concurrency deadlock path is real. The judge requires consolidating these 5 Mutex locks into a single asynchronous-aware `RwLock` to enforce safe state transitions.

5. Fix Recommendation 
struct AppStateInner {
    backend_config: BackendConfig,
    bootstrap_error: Option,
    local_backend: Option,
    standalone_jobs: HashMap,
    standalone_sequence: u64,
}

pub struct AppState(tokio::sync::RwLock);
Effort: Medium
Blast Radius: High
P1 Playwright Browser Mocking Integration Gap
tests/playwright/
1. Narrative Description

Playwright browser end-to-end integration tests are executed against a mock HTTP server inside standard browser environments, bypassing the compiled Tauri application runtime completely. All Tauri command invocations are replaced with standard web mocks. If developers make signature changes to Rust commands or Tauri bindings drift, Playwright tests still pass, leaving a major E2E integration testing blindspot.

4. Judge Ruling

Verdict: Upheld as P1 [EXISTING_DEFECT]. The mock gap leaves Tauri IPC command mappings entirely unvalidated. E2E tests must be refactored to launch the packaged binary and execute assertions within a real Tauri app frame.

5. Fix Recommendation

Reconfigure Playwright to automate compiled binaries using standard tauri-driver hooks:

// playwright.config.ts E2E Tauri configuration
import { defineConfig } from '@playwright/test';
export default defineConfig({
  testDir: './tests/playwright',
  use: {
    /* Route tests inside Tauri driver ports */
  }
});

P1 Lack of Automated Cross-Platform Packaging in CI
.github/workflows/
1. Narrative Description

The release workflow is triggered on tag push but only spawns a standard GitHub release with text bodies (`changelog.md`). It never calls `tauri build`. As a result, releases contain no compiled binaries (e.g. macOS `.dmg`, Windows `.msi`, Linux `.deb`). Developers must package binaries locally, violating basic DevSecOps release automation and security norms.

4. Judge Ruling

Verdict: Upheld as P1 [EXISTING_DEFECT]. Standard packaging automation is entirely missing. The release pipeline must run the compilation steps across a multi-platform runner matrix.

5. Fix Recommendation

Implement `tauri-apps/tauri-action` in `release.yml`:

- name: Publish Tauri App
  uses: tauri-apps/tauri-action@v2
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  with:
    tagName: v__VERSION__
    releaseName: "Stealth Lightbeacon v__VERSION__"
Effort: Medium
Blast Radius: Low
